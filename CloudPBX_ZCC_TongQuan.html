Module ZCC · CloudPBX × Zalo OA

Định tuyến cuộc gọi outbound của Agent CSOC (FTEL) qua kênh Zalo OA, tự động fallback sang SIP Trunk PSTN — kèm giao diện gửi yêu cầu cấp quyền gọi (consent) hàng loạt.

Đơn vị: SCC, CSOC / FPT Telecom Tổng đài: CloudPBX (Core FreeSWITCH) v0.1 · 2026-07-15

01 Bối cảnh nghiệp vụ

Khách hàng FTEL định danh bằng số điện thoại; Zalo định danh bằng UserID. ZCC làm cầu nối và quản lý trạng thái quyền gọi.

Tổng đài lưu khách bằng số điện thoại, còn Zalo định danh mỗi user bằng UserID (UID) trong phạm vi một OA. Module ZCC chịu trách nhiệm: chuyển đổi phone ↔ UID, quản lý quyền gọi (follow / consent), cung cấp API gửi yêu cầu cấp quyền và API quyết định định tuyến cho tổng đài. Có hai trường hợp gọi ra qua Zalo OA:

Trường hợp 1

Khách đã quan tâm (follow) OA

Gọi bằng UserID. Không cần kiểm tra quyền nếu khách chưa bỏ quan tâm và còn trong 30 ngày kể từ tương tác gần nhất.

hook event follow → API get user detail (sharing) → map phone↔UID

Trường hợp 2

Khách chưa quan tâm OA

Gọi bằng số điện thoại (84xxx). Phải gửi yêu cầu cấp quyền gọi trước; khách đồng ý thì gọi được, hiệu lực 30 ngày.

Zalo hỗ trợ gửi consent theo số thuần — không cần map UID trước

Luật định tuyến khi Agent bấm gọi. Nếu số đủ điều kiện Zalo OA → ưu tiên gọi Zalo; sau 20 giây chưa kết nối → fallback SIP Trunk PSTN. Nếu không đủ điều kiện → gọi thẳng PSTN. ZCC trả quyết định qua endpoint /resolve.

02 Giao diện gửi yêu cầu cấp quyền gọi

Màn hình web đơn giản: user được phân quyền (zcc.consent.send) nhập một hoặc nhiều số và gửi consent request tới Zalo user.

Hệ thống chuẩn hoá số về 84xxx (E.164), loại trùng/loại số sai, kiểm tra trạng thái từng số, chỉ gửi cho số chưa đủ điều kiện & chưa gửi gần đây (chống spam, tôn trọng quota Zalo), trả báo cáo theo từng số. Kiểm tra quyền ở cả frontend và backend; mọi thao tác ghi audit log.

03 Kiến trúc tổng quan

ZCC gồm frontend (màn hình consent) và backend (API, webhook receiver, worker/queue, core services, DB, cache).

04 Luồng xử lý event (Webhook)

ZCC lắng nghe event từ Zalo để cập nhật mapping/consent. Bắt buộc idempotent (dedupe theo event_id) và verify trước khi xử lý.

  • follow_oa — verify + dedupe → GET user detail (sharing) lấy phone → upsert ZALO_USER (is_follower, +30d) → invalidate callable(phone).
  • user_shared_info — cập nhật ZALO_USER.phone, củng cố mapping phone↔UID.
  • user_consent_reply — upsert CONSENT (GRANTED, +30d) + cập nhật CONSENT_REQUEST → refresh cache.
  • unfollow_oa — is_follower=false → invalidate cache; số chuyển về PSTN trừ khi còn CONSENT hợp lệ.
Vì sao cần luồng này? Nhờ webhook cập nhật liên tục, trạng thái callable luôn sẵn ở cache → endpoint /resolve trả quyết định nhanh cho việc lựa chọn kênh gọi điện.

05 Luồng gửi yêu cầu cấp quyền gọi điện (bulk)

API trả 202 Accepted ngay, worker gửi bất đồng bộ để UI không treo khi nhập nhiều số; tôn trọng quota cấp quyền của Zalo (ngày/tuần/tháng).

  • POST /consent/batch — check quyền zcc.consent.send → normalize 84xxx, loại trùng/sai → tạo CONSENT_BATCH + CONSENT_REQUEST[] → trả 202 { batch_id }.
  • Lọc & enqueue — số đủ điều kiện/vừa gửi → SKIPPED; số cần gửi (kể cả chưa follow) → enqueue sendConsent.
  • Worker — dequeue → check quota → gửi API. Thành công → SENT (+ zalo_request_id); lỗi/quota → FAILED + reason (backoff).
  • Kết quả cuối — user đồng ý → event consent_reply về webhook (mục 04); UI poll GET /consent/batch/{id}.

06 Luồng gọi điện (Zalo-first, fallback 20s)

Fallback 20s đặt tại dialplan Core FreeSWITCH (biến ZALO_CALL_TIMEOUT); ZCC chỉ trả quyết định qua /resolve và ghi call-log.
